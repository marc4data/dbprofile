## What This Project Is

**dbprofile** is an automated SQL database profiling tool. It connects to a data warehouse, runs a suite of quality checks against every column in scope, and produces:

- A self-contained HTML report with interactive navigation, charts, heatmaps, and quality scoring
- An optional Excel workbook (.xlsx) for manual EDA note-taking
- An optional JSON export for downstream tooling or re-rendering reports without re-querying

The tool supports Snowflake, BigQuery, and DuckDB connectors.

## Running the Tool

```bash
# Activate the venv
source .venv/bin/activate

# Standard run with all exports auto-named
dbprofile run --config examples/config_snowflake.yaml \
  --export-json auto --export-excel auto

# Project-based run — outputs go to <project-dir>/dq_eda/ and the
# notebook helpers (eda_helpers.py / eda_profile.py / eda_helpers_call_templates.py)
# are copied alongside on first use.
dbprofile run --config examples/config_snowflake.yaml \
  --project-dir ~/projects/portfolio_nyc_tlc \
  --export-json auto --export-excel auto

# Regenerate Excel from a previous JSON (no Snowflake connection needed)
dbprofile excel --json <dir>/dq_eda/<file>.json \
  --config examples/config_snowflake.yaml \
  --project-dir ~/projects/portfolio_nyc_tlc
```

Output files auto-name from the data source + date — e.g.
`snowflake_analytics_dbt_malex_marts_20260418.html`. With `--project-dir`,
they land in `<project-dir>/dq_eda/`. Without it, they fall back to
`./reports/`.

## Project Structure

```
dbprofile/
  cli.py                  # Click CLI — entry point, --config, --export-json, --export-excel
  config.py               # YAML loader, Pydantic models, ${ENV_VAR} resolution
  orchestrator.py          # Discovers tables/columns, fans out checks, collects results
  connectors/
    base.py               # BaseConnector ABC + Snowflake/BigQuery/DuckDB implementations
  checks/
    base.py               # CheckResult dataclass + BaseCheck ABC
    schema_audit.py       # All-null column detection, column inventory
    row_count.py          # Empty table detection, daily time series
    null_density.py       # Null % per column, sentinel values
    uniqueness.py         # Duplicate detection, identifier vs attribute classification
    numeric_distribution.py  # IQR outliers, histogram, box-whisker stats
    frequency_distribution.py  # Value frequency for low-cardinality columns
    temporal_consistency.py    # Date gaps, volume anomalies
    format_validation.py       # Regex pattern matching (email, phone, UUID, etc.)
    sample_rows.py        # Top-N sample row preview
  report/
    renderer.py           # Builds context dict + renders HTML via Jinja2
    template.html.j2      # The HTML report template (single-file output)
    excel_export.py       # openpyxl workbook writer
examples/                 # YAML config files for each connector
assets/                   # Technical specs and documentation
reports/                  # Generated output (gitignored)
```

## Key Architecture Decisions

- **Single-file HTML output** — all CSS and JS inline, only external dependency is Chart.js CDN. Portable, email-friendly.
- **Context separated from rendering** — `_build_report_context()` is a pure function returning a dict. The same context feeds the HTML template and the Excel exporter.
- **Lazy chart initialization** — chart configs registered in `<head>`, actual `new Chart()` calls deferred until the user expands a section. Keeps page load fast.
- **`autoescape=False` in Jinja2** — intentional; SVG sparklines are generated by trusted Python code and must render as markup.
- **Quality scoring** — weighted 0-100 score per table (ok=100, warn=50, critical=0 per result, weighted by check importance). Overall score weighted by table row count.

## Credentials

Credentials are loaded from `dbprofile/.env` via `python-dotenv`. The `.env` file must use full paths (no `~` expansion):

```
SNOWFLAKE_ACCOUNT=...
SNOWFLAKE_USER=...
SNOWFLAKE_PRIVATE_KEY_PATH=/Users/marcalexander/.ssh/snowflake/rsa_key.p8
SNOWFLAKE_WAREHOUSE=COMPUTE_WH
SNOWFLAKE_ROLE=TRANSFORMER
```

The `.env` file is gitignored. Never hardcode credentials in YAML configs or code.

## Coding Conventions

- Python 3.10+, type hints throughout
- Pydantic for config validation, Click for CLI
- Checks are self-contained classes inheriting `BaseCheck` — each owns its SQL and severity logic
- `CANONICAL_ORDER` in `renderer.py` defines the display order for all checks across nav, heatmap, and scorecards
- Severity palette: critical=`#f38ba8`, warn=`#f9e2af`, ok=`#a6e3a1`, info=`#89dceb`, na=`#f0f2f8`
- Template uses Catppuccin-inspired dark sidebar (`#1a1a2e`) + light main area (`#f0f2f8`)

## Common Tasks

**Add a new check:**
1. Create `dbprofile/checks/my_check.py` implementing `BaseCheck`
2. Register in `orchestrator.py` → `ALL_CHECKS`
3. Add to `CANONICAL_ORDER`, `CHECK_LABELS`, `CHECK_SHORT`, `NAV_LABELS`, `CHECK_WEIGHTS` in `renderer.py`
4. Add template block inside the `{% for cn in canonical_order %}` loop in `template.html.j2`

**Change check display order:** Edit `CANONICAL_ORDER` in `renderer.py` — one place controls nav, heatmap, scorecard, and Excel column order.

**Adjust quality score weights:** Edit `CHECK_WEIGHTS` in `renderer.py` — must sum to 1.0.
